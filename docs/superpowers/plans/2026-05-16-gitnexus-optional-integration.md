# GitNexus Optional Integration — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add GitNexus (`npm i -g gitnexus`) as an optional capability of x-skills so routing-table-driven skills (x-mindful, x-api-pentest, plus any skill that consults `mcp-toolbox.md`) prefer GitNexus's `impact` / `context` / `rename` / `route_map` / `shape_check` MCP tools when available, and fall back transparently when not.

**Architecture:** Mirror the existing optional-MCP pattern (`mcp.perplexity`, `mcp.morph`, …). One detector row in `bin/setup`, one new field in the manifest writer, one new token in `hooks/inject-capabilities.sh`, one new section in `skills/x-shared/mcp-toolbox.md`, and two surgical skill edits (x-mindful + x-api-pentest). No new skill, no new bin script, no new dependency required at runtime — every change is gated on `mcp.gitnexus` being pinned.

**Tech Stack:** bash 3.x (setup + hooks), Python 3 (manifest serializer inside setup), jq (snapshot filter), Markdown (skills + docs).

**Caveat:** GitNexus is licensed under PolyForm Noncommercial. The integration must add a one-line license note where users discover the dependency (the Feature Gates table in `CLAUDE.md` and the new section in `mcp-toolbox.md`). x-skills code itself does not embed or redistribute GitNexus.

## Prerequisites (mandatory before running any task)

1. **`jq` must be installed.** Every Task verification command below pipes through `jq`. Without it, `hooks/inject-capabilities.sh:32` exits silently (`command -v jq &>/dev/null || exit 0`) and downstream `jq` calls fail with `command not found`. Run `command -v jq >/dev/null || echo "INSTALL jq FIRST (brew install jq / apt-get install -y jq)"` once at session start.

2. **`python3` must be on PATH.** Full `./bin/setup` runs invoke a Python heredoc to serialize the capabilities manifest (`bin/setup` lines ~881-916). Tasks 1, 2, 3, 10 all require a full `./bin/setup` run; without `python3`, every one of those tasks fails before the `jq` guards matter. Run `command -v python3 >/dev/null || echo "INSTALL python3 FIRST"` once at session start.

3. **`bin/setup --check` does NOT write the manifest.** Line 927 of `bin/setup` emits `info "[check mode] Would write to $CAPS_FILE"` and skips the write. Every verifier that reads `$CAPS_FILE` (which is `${XDG_CONFIG_HOME:-$HOME/.config}/x-skills/capabilities.json` — NOT a hard-coded `~/.config/…` path; honor `XDG_CONFIG_HOME` if set) must run `./bin/setup` (full run, no `--check`) at least once after the relevant detector/manifest edits land — otherwise `jq` reads STALE state. The plan tasks below have been written to honor this constraint; do NOT silently substitute `--check` back in.

4. **Single-session safety.** The verifier subshells in Task 4 Step 3 and Task 10 Step 4 swap the live capabilities manifest. They are atomic-write-protected (write-to-tmp + `mv -f`) but you should still run those steps when no other Claude Code session is bootstrapping in parallel — concurrent SessionStart hooks could observe the force-trued manifest between the swap and the trap-restore.
