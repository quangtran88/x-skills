The model

Two phases. One commit. Every worktree.

PHASE 1 · ONCE

INSPECT — scan, profile, commit

Run once from the main checkout. Scans compose files + source + host artifacts and drafts .worktree-isolate/profile.json. User reviews interactively, then commits.

  • Detects hardcoded container_name entries
  • Detects fixed host ports
  • Detects identity-mount data dirs (${V}:${V})
  • Detects stateful singletons across 3 tiers
  • Flags cross-worktree footguns (Makefile label filters)
PHASE 2 · EVERY WORKTREE

APPLY — allocate slot, write override files

Triggered automatically by wt's post-create hook (or manually). Reads the profile, claims the next slot from the registry, emits two files into the new worktree.

  • Allocates ports: default + slot * 1000 with lsof collision scan
  • Writes compose.override.yml with !reset null + !override ports
  • Writes .env.worktree with COMPOSE_PROJECT_NAME + ports
  • Disables compose-service / env-flag singletons by default
  • Hard-blocks on host-tier singletons until ack-host-singletons

Step by step

What each step actually does — in plain English.

01

Scan the repo for everything that won't survive being run twice

Most compose stacks were written assuming one copy is running at a time. This step looks through the repo for everything that breaks that assumption.

  • Hardcoded container names — only one container can have a given name on a host. Worktree #2 collides.
  • Fixed ports — only one process can listen on port 8080. Worktree #2 collides.
  • Self-referential paths — mounts like /data:/data need an absolute host path that changes per worktree.
  • Stateful singletons — Slack bots, cron jobs, webhook receivers that should only run from one worktree at a time.
  • Cross-worktree footguns — Makefile tricks that would tear down containers from other worktrees by accident.

You confirm the findings interactively (or skip the prompts on CI). The result is a single committed profile that x-worktree-isolate uses for every future worktree.

02

The moment you make a new worktree, it kicks in

You don't have to remember to run anything. The instant you create a new worktree (via wt switch -c, git worktree add, or the /x-skills:x-worktree command), x-worktree-isolate runs in the background.

  • If the repo has no profile yet, it exits silently — your other repos aren't affected.
  • If a profile exists, it gets to work: read the profile, claim a slot, write the override files.
  • It finishes in a second or two. You don't see anything unless something needs your attention.
  • An old profile from a previous version of x-worktree-isolate? It refuses with a clear message and asks you to migrate — never silently misbehaves.
03

Give this worktree its own number and its own ports

Every worktree of the same repo gets a unique "slot" number — 0, 1, 2, and so on. The slot decides which ports this worktree uses. The math is simple and predictable: if Postgres normally runs on 5432, slot 1 puts it on 6432, slot 2 on 7432, and so on.

  • Same worktree, same ports — every time. Re-running the setup doesn't shuffle anything.
  • The numbers are checked against what's actually free on your machine before being committed — if some other process is squatting on a port, the next slot up is used instead.
  • Two worktrees can't claim the same slot — a quick lock makes the first one win.
  • Removing a worktree frees its slot for the next one.
04

Write the override files — and don't touch anything else

x-worktree-isolate never edits your compose.yml, your .env, or your Makefile. It only writes two new files in the worktree, both ignored by git:

  • An override that strips out the conflicting container names, reassigns the ports, and disables the singletons that shouldn't run here.
  • An env file with a unique project name, the new port numbers, the worktree's data directory, and any "feature off" flags the app reads.
  • Both files are auto-generated, carry a "DO NOT EDIT" header, and are removed cleanly when you remove the worktree.

Compose picks both files up automatically — you launch the way you always did, and everything just works.

05

For things that live on your machine — refuse, ask for permission

Some singletons aren't in compose at all — they're things like a host crontab or a systemd timer running on your laptop. x-worktree-isolate doesn't have a safe way to disable those (touching your machine's cron would be invasive and risky).

  • So instead, it stops and names the file that worried it.
  • You handle it yourself — comment out the cron line, stop the systemd timer, or leave it running if this is the worktree that should own it.
  • Then you tell x-worktree-isolate "I've handled it" with one command, and the setup proceeds.

This is the only place the tool isn't fully automatic. The trade-off is intentional: better to ask than to quietly mess with your machine's state.

Emitted files

Exactly two files per worktree.

Plus a state snapshot. Nothing else mutated. The launch command is printed for you to run.

compose.override.yml
# auto-generated by x-worktree-isolate v0.2 · DO NOT EDIT
services:
  api:
    container_name: !reset null
    ports: !override
      - "${API_PORT}:3000"

  postgres:
    container_name: !reset null
    ports: !override
      - "${POSTGRES_PORT}:5432"

  slack-listener:
    profiles: [xwi-disabled]
.env.worktree
# auto-generated · slot 2
COMPOSE_PROJECT_NAME=x-skills-feat-x
API_PORT=5000
POSTGRES_PORT=17432
REDIS_PORT=18000
DATA_DIR=/abs/path/wt-2/data

# tier-2 env-flag singletons
SCHEDULER_ENABLED=false
SLACK_BOT_ENABLED=false

# launch command (printed by apply):
# docker compose --env-file .env --env-file .env.worktree up
Subcommands

One CLI. Sixteen verbs.

All routed through dispatch.sh. Hooks call apply --quiet --if-profile-exists automatically; everything else is manual.

Design principles

Why it works the way it does.

P1

Never edits your code

Your compose.yml, your .env, and your Makefile are read-only. All changes go into two new files alongside them.

P2

Asks rather than guesses on your machine

Anything that lives outside the repo (crontab, systemd) is flagged, not touched. You decide what to do.

P3

Predictable port math

Worktree #N gets ports offset by N. Same worktree always gets the same ports. No surprises after a reboot.

P4

Only one safe way to strip a setting

Docker Compose silently keeps the old value if you set it to blank. The override uses the one syntax that actually clears it.

P5

One file holds the truth

What ports got allocated, what the project name is, where the data lives — it's all in one small file. Other tools read from there.

P6

Singletons start disabled

Every new worktree starts with bots, schedulers, and webhook receivers turned off. You explicitly turn them on in the worktree that should own them.

Get started

One profile per repo. Forever.

~/your-repo
# 1. From main checkout: scan + write profile + commit
$ /x-skills:x-worktree-isolate init
$ git add .worktree-isolate/profile.json .gitignore
$ git commit -m "chore: add worktree-isolate profile"

# 2. Create a new worktree (auto-applies via post-create hook)
$ wt switch -c feat/x
✓ x-worktree-isolate: slot 1 claimed · ports 5001/15432/16000 · 0 host singletons

# 3. Launch (single command — base + override merged)
$ docker compose --env-file .env --env-file .env.worktree up

# 4. Inspect what's claimed
$ /x-skills:x-worktree-isolate list
