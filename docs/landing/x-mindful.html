The taxonomy

Five things AI does silently.

Every extracted item lands in exactly one category. No code-level findings — the human reviews direction, not implementation.

TRADEOFF

A fork in the road. AI picked side X without naming the cost on the other axis.

sync vs async · monolith vs split · build vs buy · one-way vs two-way door

ASSUMPTION

A load-bearing claim AI silently relied on without surfacing.

scale · NFRs · environment · human capacity · trust boundary

BLIND-SPOT

What's NOT in the plan that a senior would expect.

failure modes · rollback · observability · runbook · threat model · owner

SHAPE

The solution shape doesn't fit constraints — over- or under-engineered.

modular mirage · microservice for nothing · missing idempotency · no retries

FUTURE-DEBT

What gets harder later. Paint-into-corner choices and curves that grow.

vendor lock-in · cost curve · org-fit assumption that won't hold
Workflow

Five phases. One decision envelope.

The gate

Four actions per item.

Phase 4 walks the ranked queue. For each item the user picks exactly one of:

✓ CONFIRM

proceed as written

The item lands in the envelope's Confirmed section. x-do Mode A executes it unchanged.

✎ MODIFY

revise the direction

User supplies a new direction in prose. Lands in Modified; x-do revises the plan before executing.

✗ REJECT

drop from plan

User gives one-line reason. Item lands in Rejected and is removed from x-do's queue.

› SKIP

revisit later

Item lands in Skipped — surfaced again on the next x-mindful run against the same plan.

Step by step

What each phase actually does — in plain English.

01

Find the plan and check it's worth reviewing

You can give x-mindful the plan in three ways: point at a file, paste the text right in, or hand off from another skill that just produced one. The first step normalizes all three into the same shape.

  • If you point at a file, x-mindful reads it.
  • If you paste content, it's tagged so we can refer back to it.
  • If another skill (like x-do) is calling in, the plan content rides along in the handoff message.
  • If there's no actual plan content, x-mindful stops and points you to x-do — there's nothing to review yet.
  • The plan's size (short / medium / long) decides which AI helper does the deep read in the next step.
02

Read the plan and pull out the senior-engineer concerns

An AI helper reads the plan five times, once per category. Each pass asks a different question: "what tradeoff was made silently?", "what assumption is load-bearing?", "what's missing that a senior would expect?", and so on.

  • Short plans → one quick pass by Claude directly.
  • Medium plans → Gemini if available, otherwise Claude with the plan broken into chunks.
  • Long plans (a whole RFC, design doc, or PRD) → a deep-reading agent that can hold the whole thing at once.
  • Hard rule: any finding that mentions a function name, file path, or variable is dropped or rewritten. Those belong in code review (x-review), not architectural review.
  • When code-graph data is available, the actual blast-radius of touching a piece of code (depth-1 = will break, depth-2 = likely affected) replaces guesswork.
03

Rank by what really matters — and drop the noise

The extract step usually finds more items than a human wants to walk through. Step 3 throws away anything trivial and orders the rest by what would actually keep a senior engineer up at night:

  • How bad if it goes wrong? — Critical / High / Medium / Low
  • How much of the system does it touch? — one file, one team, the whole org
  • How hard is it to undo? — Renaming an internal helper tomorrow is cheap. Switching from sync to async, or breaking a public API, is not.

The output is a short, ordered list. The most consequential items come first so you can stop early if you've already heard enough.

04

Walk you through one item at a time

This is the core of the skill. First, a 5-bullet summary of the plan at the architect level — so you can see what AI is proposing before diving into the concerns.

Then x-mindful walks the ranked list one item at a time. Each item shows: what category it falls in, how serious it is, what part of the system it touches, and the question the senior engineer would ask. You pick: confirm it, modify it (give a new direction in your own words), reject it (drop from the plan with a reason), or skip it for now.

Your decisions live in the session — nothing is written to disk unless you ask. The conversation is the artifact.

05

Hand off a clear, copy-pastable decision summary

At the end, x-mindful prints a short markdown summary with four sections: what you confirmed, what you modified (with your direction quoted), what you rejected (with your reason), and what you skipped for later.

  • Confirmed → the build step uses the original plan for these items.
  • Modified → the plan is revised to match your direction before any code is written.
  • Rejected → these items are removed from the plan entirely.
  • Skipped → tracked as follow-ups; they'll come back on the next review.
  • If you want a permanent record, ask x-mindful to save it. Otherwise the summary lives in the chat for the rest of the session.

Anti-triggers

When NOT to use x-mindful.

x-mindful gates work before it starts. If the request is closer to one of these, route there.

User intentRoute to
"Review this code / PR / diff" — judging existing workx-review
"Walk me through and teach me this doc" — comprehensionx-guide
"Investigate / find how X works" — open researchx-research

Design principles

Why it works the way it does.

P1

You decide direction, AI handles the rest

No talk of function names, file paths, or syntax. Those AI does fine on its own — humans should be weighing in on direction.

P2

One item at a time

x-mindful never dumps a wall of concerns. You see one, decide, move on. You set the pace.

P3

Worry more about what's hard to undo

Choosing a name can be fixed tomorrow. Switching to event-driven and changing a public API cannot. The ranking reflects that.

P4

No files unless you ask

The review lives in the chat. You can save the summary if you want a record — otherwise nothing is written.

P5

Before, not after

x-review reads code after it's written. x-mindful reads the plan before it's built. Same goal, different timing.

P6

The summary is the contract

The next step (usually x-do) reads your decisions from the summary, not from anywhere else. Clean handoff, no surprises.
