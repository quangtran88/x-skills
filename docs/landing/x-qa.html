Step by step

What each step actually does — in plain English.

No jargon. The decisions, the inputs, the outputs, and what x-qa refuses to do.

01

Pick the helpers — who runs the simple tests, who runs the hard ones

Before anything else, x-qa figures out which AI helpers are available in this session and picks two: one cheap helper for quick HTTP checks, one stronger helper for multi-step flows. The choice is locked in once so every test that follows knows where to go.

  • Cheap-and-fast: Gemini Flash if available; otherwise a smaller Claude.
  • Smarter-but-slower: a specialized Claude tester agent (or a Gemini equivalent if that's all that's pinned).
  • Refuses if you're not in a git repo — every artifact this skill writes is repo-scoped.
02

Figure out what to test

You can hand x-qa anything — a PR number, a feature name, a file, a folder, a spec document, or just a sentence in plain English. A small set of rules decides which kind it is, so the rest of the pipeline knows where to look.

  • nothing typed → test what's new on the current branch
  • "PR #482" → test what that PR changes
  • a feature name → smoke-test that one entry point
  • a markdown spec → read the doc, scan the code it implies
  • a file or folder → walk the code around it
  • a sentence ("test password reset") → search the repo for related code

When the guess is uncertain, x-qa asks one short question before continuing — never silent guessing.

03

Walk the codebase to bound the scope

When you give it a spec document, a folder, or a plain-English sentence, x-qa doesn't try to test the whole app. It runs a quick "scout" pass first: read the input, walk the related code, list out the URLs the change actually touches plus the edge cases that come to mind.

  • Bounded: scout stops at 20 URLs and 40 edge cases so it can't run away in a huge repo.
  • Open questions the scout couldn't answer get carried straight into the final report — humans see them flagged.
  • For a specific PR or named feature, this step is skipped — the scope is already obvious.
04

Build the test list — reuse what worked, write what's new

A test plan is just a list: "for this feature, here are the requests we should send and what we expect back." The planner doesn't start from scratch. It looks in your team's library of past tests first — anything that's been passing for a while gets reused as-is. Only the gaps get written fresh.

① CHAINS FIRST

A chain is a multi-step scenario like "log in → add to cart → check out" that's worked end-to-end on the last several runs. If the change you're testing touches any of the steps, the whole chain comes back in.

why first: chains catch the most subtle bugs — the kind that show up only when steps run in sequence.

② SINGLE TESTS NEXT

For each URL the change touches, the planner asks: "do we already have proven happy-path / error / edge tests for this?" Anything that's been green long enough gets pulled in as-is.

why second: reuse before rewriting. Tests that have run hundreds of times are far more trustworthy than tests just drafted by an AI.

③ WRITE THE REST

Whatever still has no coverage — a new URL, a missing error scenario — the AI drafts fresh. These are tagged generated, run alongside the reused ones, and become candidates to enter the library themselves if they keep passing.

why last: minting is the slow, error-prone path. Use it only when needed.

REFUSAL RULES — THE PLANNER WILL NOT EMIT IF…
✗
A test references a URL the app doesn't have

The planner cross-checks every URL against the live map of the app. AI-invented endpoints are dropped with a clear error — no hallucinations sneak into the plan.

✗
Any URL in scope is missing a happy-path test

Every URL the change touches must have at least one "does this work at all?" test. Error tests and edge tests are encouraged but not required.

⚠
A library test targets a URL that was renamed

The planner skips the stale one and flags it for human review. The library entry isn't auto-deleted — the URL may have just moved.

⚠
A test has a duplicate name

If a fresh test would collide with one already in the library, the planner reuses the library version (or suffixes the new one) so names stay stable across runs.

After this step the planner prints a one-line summary so you can see the mix at a glance — e.g. 14 reused · 9 new · 2 chained flows. High reuse means the library is paying off; lots of new means the change is exploring fresh territory.

05

Start the app and wait for it to answer

Tests need a running app. x-qa starts the service the way you'd start it normally (usually docker compose up), then politely waits — pinging a health URL until the app says "I'm ready" or a timeout fires.

  • Already running on staging? Point x-qa at the live URL and the launch step is skipped.
  • Running two PRs in parallel on the same machine? Each worktree gets its own ports and project name so containers don't collide. See x-worktree-isolate.
  • Between tests the database can either reset (safer, slower) or carry over (faster, riskier) — you choose in the profile.
06

Decide which tests are quick vs which need a smarter helper

Tests come in two shapes:

  • Quick checks — one request, simple answer ("does this URL return 200?"). The cheap helper handles these.
  • Stateful flows — log in, get a token, use the token on a different request, watch what changes in the database. The smarter helper handles these.

When in doubt, x-qa treats a test as stateful. The reasoning: spending a little more on a test that turned out to be simple costs you 10×. But running a complex test on the cheap helper and getting a wrong answer costs you a missed bug — which is much worse.

07

Run tests in parallel — respecting what depends on what

Some tests depend on others — you can't "add to cart" until you've logged in. x-qa figures out the order and runs as many tests in parallel as possible.

  • Wave 1: every test with no dependencies runs at the same time (login, seed data, ping checks).
  • Wave 2: only starts after every wave-1 test finishes. Tests that needed login go now.
  • And so on. Up to 8 tests run in parallel by default.
  • If a test fails, anything that depended on it is marked skipped — not failed. The skipped tests don't drag down the overall verdict, but you see them in the report.
  • If two tests need each other (a cycle), x-qa refuses the plan with a clear message rather than guess.
08

Every test writes a result card

Each test, simple or complex, produces a small report card with the same fields: did it pass? how long did it take? what did the server actually return? if it failed, what failed?

  • Pass / fail / flaky-recovered — those are the only verdicts.
  • The raw response is kept so a human (or x-bugfix) can look at exactly what came back.
  • If a test produces a malformed card (e.g. the helper wrapped its JSON in extra text), x-qa treats it as a failure and quarantines the raw output for review.
  • If a test silently produces nothing, x-qa marks it as a failure rather than pretending it passed.
09

Handle flaky tests — retry, but never quietly

Some tests fail because the network blinked or a container was still starting. x-qa retries a couple of times before giving up.

  • A test that fails once then passes is marked flaky-recovered — not a pure pass, not a fail.
  • Flaky-recovered doesn't block the run — only true failures do.
  • Flaky-recovered also doesn't earn trust — it can't graduate into the team's library based on a retry.
  • Every recovered test shows up in the final report with how many attempts it took, so you can see patterns over time.
  • You can set a max tolerated flaky percentage — above that, x-qa fails the run on principle.
10

Roll everything up — and remember what we saw

Every test's report card gets folded into a single human-readable report and a machine-readable summary. At the same time, x-qa updates its memory:

  • Per URL: which status codes we've seen, how fast it usually responds (median + slow-tail), when it was last tested.
  • Per run: a one-line log entry capturing what passed, what failed, and which chains worked end-to-end.
  • Drift signals: x-qa notices when a URL starts behaving differently (slower, different errors) — but only flags it for review, never fails the run automatically. Observation, not enforcement.
11

New tests earn their place in the library

A fresh test that passes once is interesting. A fresh test that passes three runs in a row — exactly the same way, no failures in between — is trustworthy. That's when it joins the team's shared library.

  • Only true passes count. A flaky-recovered pass doesn't add to the streak.
  • The test has to be unchanged across those runs. If the planner kept rolling slightly different versions, none of them counted as the same test.
  • One failure resets the streak to zero. Trust is rebuilt from scratch, not patched.
  • Chains of tests (login → cart → pay) only graduate after every single step has graduated on its own first. Expect a one-run delay.
  • Tests that go bad later stay in the library — a human reviews them. x-qa never silently deletes proven tests; it surfaces them for the team to decide.

This whole system is the reason runs get faster over time: the more your team uses x-qa, the more of the test plan comes from proven tests instead of fresh AI drafts.

Design principles

Why it works the way it does.

P1

Works with whatever AI you have

Gemini, Claude, or both — x-qa adapts at startup. Nothing in the test logic is tied to one model.

P2

When in doubt, use the smarter helper

Spending more on a test is cheap. Missing a bug because the cheap helper got confused is expensive.

P3

Refuse, don't guess

Test a URL the app doesn't have? Two tests need each other? Stale data? Stop and say so — never silently guess.

P4

Reuse before writing

The planner always checks the team's library first. Fresh tests are written only for the gaps.

P5

Notice, don't enforce

Slowdowns, drift, new error patterns — all flagged for humans, never auto-failed. You stay in control.

P6

Read every result before judging

x-qa always waits for every parallel test to finish and reads its card. No silent successes from missed output.
