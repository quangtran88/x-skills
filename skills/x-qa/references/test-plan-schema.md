# TEST_PLAN.md Schema

YAML format. Generated by `generate` or `run`. Consumed by dispatch + aggregate.

## Top-level

| Field | Type | Required | Notes |
|---|---|---|---|
| `feature` | string | yes | Human-readable feature name. |
| `entry_point` | string | yes | Must match `profile.entry_points[].name`. |
| `acceptance` | string[] | yes | Acceptance criteria, one per bullet. |
| `qa_strategy` | QAStrategy | yes | Service launch + URL config. |
| `test_cases` | TestCase[] | yes | At least 1. |
| `gates` | Gate[] | no | Optional declarative thresholds, see `references/quality-gates.md`. |

## QAStrategy

| Field | Type | Required | Notes |
|---|---|---|---|
| `service_launch` | string | no | Override `profile.entry_points[].launch.command`. Rare. |
| `base_url_var` | string | no | Env var holding base URL. Default resolved from profile. |
| `health_check` | string | no | Override profile health probe. |

## TestCase

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | string (slug) | yes | Stable across runs. e.g. `tc-avatar-happy-jpeg`. Reused from `kb/index.json.cases` when the corpus already has a green case for this endpoint+category; otherwise minted per `kb-schema.md § Stable ID Derivation`. |
| `provenance` | enum | no | `corpus` \| `corpus-flow` \| `generated`. Set by the planner. `corpus` = pulled verbatim from `kb/cases/`; `corpus-flow` = pulled as part of a `from_flow` expansion; `generated` = net-new this run. Defaults to `generated` when absent. |
| `from_flow` | string | no | Flow ID this case was expanded from (e.g. `fl-user-signup-then-avatar`). Set only when `provenance == corpus-flow`. |
| `category` | enum | yes | `happy` \| `edge` \| `error` \| `auth` \| `concurrency` \| `regression`. |
| `complexity` | enum | yes | `simple` \| `complex`. Auto-set by classifier; user override allowed. |
| `description` | string | yes | Human-readable. |
| `setup` | string | no | Setup command/fixture name. |
| `request` | Request | http only | HTTP request spec. |
| `assertions` | Assertion[] | yes | One per check. |
| `teardown` | string | no | Cleanup command. |
| `timeout_ms` | int | no | Per-case timeout. Default 30000. |
| `depends_on` | string[] | no | List of `test_case.id` values that MUST pass before this case runs. Empty/absent = no deps (parallel-eligible). Cycles refused by dispatcher. |
| `parallel_group` | string | no | Reserved hint for future use. The v1 dispatcher infers waves from `depends_on` topology only and ignores this field. Planners may emit it for forward compatibility. |
| `covers` | string[] | no | Obligation ids this case satisfies (`field:…`/`inv:…`/`trans:…`/`xtrans:…`/`fmode:…` per `references/scout-prompt.md § Obligations`). Read by `scripts/coverage-check.sh`. A case may cover several; an obligation may be covered by several. |

## Request (http)

| Field | Type | Required | Notes |
|---|---|---|---|
| `method` | enum | yes | `GET`/`POST`/`PUT`/`PATCH`/`DELETE`. |
| `path` | string | yes | URL path appended to base_url. |
| `headers` | object | no | Header map. |
| `body` | any | no | JSON body. |
| `query` | object | no | Query params. |

## Assertion

| Field | Type | Required | Notes |
|---|---|---|---|
| `kind` | enum | yes | `status` \| `body-jsonpath` \| `body-regex` \| `header` \| `latency-ms` \| `custom`. |
| `expr` | string | yes | Expression depending on kind (e.g. `$.id` for jsonpath). |
| `op` | enum | yes | `eq` \| `neq` \| `contains` \| `matches` \| `lt` \| `gt`. |
| `value` | any | yes | Expected value. |

## Sequencing Semantics

Dispatcher behavior:
1. Build a DAG: nodes = `test_cases[].id`, edges = `depends_on`.
2. Refuse the plan if the DAG has a cycle (`scripts/lib/topo-order.sh` returns non-zero).
3. Compute waves via Kahn's algorithm. All cases in a wave dispatch in
   parallel (capped at `--max-bg`). Wave N+1 starts after every case in
   wave N reaches a terminal verdict.
4. A `fail` in wave N skips downstream dependents (mark `skipped`,
   not `fail`, in QA_REPORT.md). Independent branches continue.

If `depends_on` is empty across all cases, behavior matches v1 (flat
parallel fanout). Backwards compatible.

## Required Coverage Obligations (planner contract — obligation-gated)

Coverage is enforced against `scope.json.obligations[]`, not category presence:

1. **Every `severity: required` obligation** is satisfied by ≥1 case whose
   `covers[]` lists that obligation id. `scripts/coverage-check.sh` refuses the
   plan otherwise (Run Phase 7.5).
2. **Happy/edge cases assert the outcome, not just the status.** For every
   `inv:<slug>` obligation, a case MUST assert the invariant on the *success*
   response or resulting state (a `body-jsonpath`/`custom` assertion) — catching
   the "false case" (200 with a wrong result). A case that only asserts
   `status == 2xx` does NOT satisfy an `inv:` obligation.
3. **Still emit ≥1 `happy` case per reachable endpoint** (unchanged baseline).
4. `recommended` obligations are encouraged but never block the gate.

## Corpus Reuse (planner contract — KB-aware)

When `kb/index.json` exists and the run is not `--no-kb`, the planner
MUST consult the corpus BEFORE minting new case bodies. Order of
operations:

1. **Match flows first.** For each `kb/flows/fl-*.yaml` whose
   `endpoint_set` is a subset of the in-scope endpoints AND whose chain
   of cases is not quarantined, splice the flow's `steps[]` into
   `test_cases[]` with `provenance: corpus-flow` and
   `from_flow: <flow-id>`. Preserve `depends_on` from the flow body.

2. **Then match individual cases.** For each in-scope
   `(endpoint, category)` pair, look up `kb/index.json.cases` entries
   matching that endpoint + category that are NOT quarantined and NOT
   already pulled in by a flow. Splice them with `provenance: corpus`.

3. **Then generate net-new for gaps.** Only after steps 1 + 2, the
   planner mints `provenance: generated` cases for any
   `(endpoint, category)` pair that has no corpus coverage AND is
   required by `Required Coverage` above.

4. **Stable IDs.** A generated case's ID is derived per
   `kb-schema.md § Stable ID Derivation`. If a generated case's body
   matches an existing (quarantined) corpus entry's checksum, the
   planner reuses that ID and warns. If the body differs and the slug
   collides, suffix `-2`, `-3`, …

5. **Drift refusal.** If a corpus case references an endpoint absent
   from the current `profile.json` catalog (endpoint rename, deletion),
   the planner skips it and emits a `corpus-stale` warning. The case
   is NOT auto-demoted from a single run — demotion is the curation
   layer's job (see `kb-curation.md`).

The planner MUST emit a one-line summary on stderr:
`KB_REUSED=<n> KB_GENERATED=<m> KB_FLOWS=<f>` — the orchestrator
folds these counters into the run envelope.

## User Hints Block (Convention)

When the run was invoked with prose intent (`intent.json.intent == "prose"`) or with free-form `--service`/`--branch`/`--pr` plus an inline description, the planner prompt MUST include the user's raw text as a dedicated `## User Hints` markdown block, NOT mixed into the directive prose.

Placement: immediately before the `## Task` block, immediately after any code-context blocks. Format:

```markdown
## User Hints (prioritization guidance, may be empty)

> {{intent.json.resolved.prose}}

## Task
…
```

**Rationale.** Verbatim isolation prevents the LLM from treating user hints as instructions to override the planner's structured contract. The leading `>` blockquote marker makes the boundary visually unambiguous in transcripts.

**Empty case.** When no prose is present, emit:

```markdown
## User Hints (prioritization guidance, may be empty)

> (none)
```

Do NOT omit the block — its presence is part of the prompt's stable shape.

## Gates

Optional plan-level quality gates. Schema and evaluation rules in `references/quality-gates.md`. When the plan declares a `gates:` block, it **fully replaces** profile defaults — no merging.

```yaml
gates:
  - { metric: tests.passRate, threshold: 100, blocking: true }
  - { metric: tests.flakyRate, max: 5, blocking: false }
```

## Optional Step Mode: `ai_action` / `ai_assertion` (forward-compat, deferred)

A TEST_PLAN step MAY declare `mode: ai_fallback` to express "this step is best satisfied by an LLM-driven action when literal selectors/paths fail." Two action shapes are reserved:

```yaml
steps:
  - mode: ai_fallback
    type: ai_action
    goal: "dismiss the cookie banner if present"
    optional: true              # true → skip step if AI declines (default false)
  - mode: ai_fallback
    type: ai_assertion
    goal: "the order confirmation page shows a green checkmark"
```

**Status.** Schema-only in v1. The dispatcher rejects `mode: ai_fallback` steps with a clear error message pointing to this section, until the browser/selector phase activates the tier. This reservation prevents future schema collisions.

**Routing once wired.** `mode: ai_fallback` steps force the complex (claude) runner per `classification-rules.md` — the cheap (gemini) runner skips them. Runner consumes the step under the `FallbackResponse` contract from `fallback-contract.md`.
