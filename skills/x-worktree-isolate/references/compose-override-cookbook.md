# compose-override-cookbook.md

Why `!reset null` is the only working incantation.

## The problem

Compose v2 merges `compose.yml` and `compose.override.yml` (auto-loaded if present in cwd). For most fields, the override file's value replaces the base. But for `container_name`, several "obvious" override forms silently fail to remove the base value.

## Empirical verification (Compose v2.40.3)

```
# /tmp/compose-null-test/compose.yml
services:
  web:
    image: nginx
    container_name: my-web
    ports:
      - "8080:80"
```

| `compose.override.yml` | `docker compose config` shows |
|---|---|
| `services: {web: {container_name: !reset null}}` | **container_name removed.** Compose auto-generates `<project>-web-1`. |
| `services: {web: {container_name: null}}` | `container_name: my-web` (base retained, silent). |
| `services: {web: {container_name: ""}}` | Schema validation error: empty string. |
| `services: {web: {}}` (no container_name key) | `container_name: my-web` (base retained, silent). |
| `services: {web: {container_name: "different"}}` | `container_name: different` (override replaces, but doesn't remove). |

## The other tag we use

| Tag | Effect |
|---|---|
| `ports: !override [...]` | Replaces the entire ports list. Without `!override`, Compose **appends** override ports to base ports — leaving the hardcoded `127.0.0.1:18789:...` next to `127.0.0.1:${GATEWAY_PORT}:18789`, defeating the purpose. |

## Generated override shape

```yaml
# Auto-generated by x-worktree-isolate — do not edit by hand.
services:
  oneclaw-multiagent:
    container_name: !reset null
    ports: !override
      - "127.0.0.1:${GATEWAY_PORT}:18789"
  oneclaw-dashboard:
    container_name: !reset null
    ports: !override
      - "127.0.0.1:${DASHBOARD_PORT}:3000"
  # ... one block per profile.services_to_strip entry — keyed by SERVICE name
  #     (Compose merges by service key, NOT by container_name).
```

## What this skill does NOT touch

- The base `compose.yml` — never edited.
- Image tags — content-addressed, safe to share.
- Env values inside services — even when they reference a stripped container_name, the skill only flags them in `service_dns_references` and lets the user decide.
- The user's `.env` — never read, never written, never symlinked.

## Source

- [Compose merge spec — `!reset` and `!override`](https://docs.docker.com/reference/compose-file/merge/) — supported since Compose v2.20+, broadly stable in v2.24+.
- Stacked `--env-file` (right-most wins) — supported since Compose v2.16 (Feb 2023).
